# 错误复盘

本文件记录 AI 在本项目中被用户纠正过的错误，以及以后必须执行的防复发动作。
不要只写“下次认真”，每条都要能指导后续验收。

## 2026-07-12 · 不能把解析辅助图误判成题干图形依赖

- 场景：数量关系剩余队列纠偏时，我一度把“解析页中出现如图、坐标轴、辅助示意”等文字也纳入图形依赖门禁，导致 approved_seed 从 535 错退到 502。
- 我做错了什么：把“题干/选项需要原图才能作答”和“解析里为了讲解画了辅助图”混为一谈。前者必须留后台裁原图，后者不应影响纯文字题进入前台。
- 根因：图形依赖判断过度泛化，直接扫描解析全文，没有区分用户作答所需信息和解析讲解所用信息。
- 以后必须这样做：图形依赖门禁默认只看题干和选项；若题本缺页或 OCR 严重缺块，只能通过主 Agent 肉眼确认后的窄范围名单补充，不能用解析全文里的“如图/坐标轴”全局拦截。
- 可验证的防复发动作：修改图形门禁后必须跑 `npm run quantity:pipeline`，确认 approved 数量没有因误伤异常下降；若下降，必须列出被新增 `image_dependent_question` 的题并抽查是否真为题干/选项依赖图。

## 2026-07-10 · 回看原解析页时不能只盯答案，能看见的选项也要顺手核对

- 场景：数量关系题库答案从 OCR 审计层升级为原页确认时，我连续按“亲眼看到参考答案”记录了若干题，但用户提醒“选项你不能顺便看了吗”。
- 我做错了什么：流程过窄，只把原页回看当作答案确认，没有把同页可见的题干和 A/B/C/D 选项顺手和入库数据对照。这样虽然降低了错答案风险，但仍可能留下“答案对、选项 OCR 有细小错误”的隐患。
- 根因：把“答案来源升级”和“题目结构上线质量”拆得太机械；实际生产中，既然已经打开原图，边际成本最低的质量动作就是同步检查可见题干/选项。
- 以后必须这样做：后续每次打开原题页或解析页，若页面可见完整题干和 A/B/C/D，必须同时核对入库 stem/options；若只确认答案，记录必须明确只升级答案来源，不得暗示选项也已人工确认。
- 可验证的防复发动作：新增修复记录时，`review_note` 要写清“仅确认答案”或“答案与可见选项均核对”；若要把选项升为人工确认，必须记录来源页并跑 `node tools/quantity-bank-validate-approved.mjs` 和 `node tools/quantity-bank-audit-approved-answers.mjs`。

## 2026-07-10 · 并行复核只能做报警器，错答修正必须亲眼看原页

- 场景：数量关系题库 461 道通过结构门禁后，用户明确要求“最保险的就是直接去看答案”“一定要亲眼确认”。并行 worker 在 231 道 OCR 证据审计题中筛出 15 道答案冲突，主 Agent 逐页打开解析原图确认后才写入修复。
- 我做错了什么：此前容易把“有答案审计脚本/并行复核结果”当成接近最终事实，但题库答案属于高风险数据，不能把 worker 的 OCR/文本判断直接当成上线依据。
- 根因：并行复核能提升发现问题的速度，却不能解决来源可信度问题。真正的权威来源仍是解析 PDF 原页的参考答案和同题解析；OCR 文本、手算、worker 摘要都只能辅助定位。
- 以后必须这样做：对已发现的错答冲突，必须主 Agent 或明确人工视觉逐页看原解析页后再写入 `verified_repairs.json`。未亲眼确认的一致题不得批量升级为 `manual_or_original_page_verified`。
- 可验证的防复发动作：答案回灌脚本必须支持只导入已确认 mismatch，例如 `--mismatches-only`；每条错答修复必须写 `manual_analysis_page_answer_correction` 来源说明；跑完 pipeline 后必须列出哪些题留前台、哪些因题干/解析脏退后台。

## 2026-07-10 · 同题多条修复必须合并，不能让后置答案修复覆盖题干修复

- 场景：15 道错答修正后，`05_q09`、`22_q01`、`22_q02` 临时退回后台。排查发现它们早已有 `original_question_page_visual_review` 的干净题干/选项修复，但后写入的 `manual_analysis_page_answer_correction` 只包含答案字段，清洗脚本用 Map 按 id 取最后一条，导致早先题干/选项修复被覆盖。
- 我做错了什么：最初假设一题只有一条最终修复记录，没有把“先修题干/选项、后修答案、再修解析/媒体”的长期迭代场景设计进去。
- 根因：修复层是增量事实日志，不是单条覆盖表。题库修复后期，同一道题很可能经过多次独立校正；如果按 id 覆盖，只会保留最后一次局部修复，丢失之前已经确认过的字段。
- 以后必须这样做：`verified_repairs.json` 中同一题的多条 repair 必须按顺序合并字段；后置字段只能覆盖同名字段，不能删除前序已经验证的 stem/options/media/topic。source 也要保留能证明题干/选项来源的类型。
- 可验证的防复发动作：清洗后检查 `05_q09`、`22_q01`、`22_q02` 必须同时保留干净题干和最新正确答案；`npm run quantity:pipeline` 应输出 461/139，且 15 道错答专项检查全部在 approved 中为 `manual_or_original_page_verified`。

## 2026-07-10 · approved 总数不等于答案零风险

- 场景：数量关系题库推进到 461 道后，用户担心“之前的 OCR 其实也有问题，答案错了很严重”。复查发现 461 道中只有 159 道属于人工/原页确认，71 道为严格推导，231 道仍属于 OCR 证据审计通过但建议正式上线前回看原解析页。
- 我做错了什么：前面容易用 `approved_seed` 或“高置信题”这个总数表达进度，虽然有答案审计脚本，但没有把答案来源风险在前台和状态文档中直接暴露出来。
- 根因：结构门禁、答案高置信、答案人工确认是三层不同概念。OCR 提取到清晰参考答案并通过正则证据，不等于人工回看原解析页确认。
- 以后必须这样做：题库类项目必须同时报告总题数、结构门禁通过数、人工/原页答案确认数、严格推导数、OCR 证据审计数和需要原页复核数。不能把总数说成“全部可正式上线”。
- 可验证的防复发动作：导出给前台的题必须带 `answer_audit_tier`；CI 必须检查答案分层总数等于 approved 总数；前台文案不得只显示“高置信题 N 道”，必须暴露需要原页复核的数量。

## 2026-07-10 · 自动重配成功不等于题目可以上线

- 场景：数量关系 600 题修复后期，部分题通过自动重配获得了题干、选项、答案和解析，看起来字段齐全；但回看 `quantity_hs13_set60_q09` 原题页后发现，真实题目是图形依赖的几何最短路径题，自动重配内容实际串到了别的文字题。
- 我做错了什么：前期容易把“字段齐全 + 有答案 + 有解析”当成可继续推进的候选，若不回看原题页，就可能把错题型、错解析、错答案放进前台训练。
- 根因：OCR 切块和相似度重配只能降低人工定位成本，不能替代同题校验；尤其是图形题、跨页题、题号残片、解析串页时，自动匹配会产生“格式完整但语义错误”的假阳性。
- 以后必须这样做：自动重配题只能作为后台候选；进入 `approved_seed` 前必须回看题本原页和解析页，确认题干、选项、答案、解析是同一道题。发现图形依赖但没有原 PDF 局部裁剪资产时，必须继续留后台。
- 可验证的防复发动作：`analysis_auto_rematched_needs_review`、`analysis_stem_mismatch`、`image_dependent_question` 任一标签存在时，不得进入前台；修复记录必须写清楚核验的题本页和解析页，跑 `npm run quantity:pipeline` 后 approved 中这些高风险标签数量必须为 0。

## 2026-07-10 · 低置信 OCR 答案可能是错答案，不能批量升级

- 场景：数量关系 600 题修复过程中，`quantity_hs13_set21_q03` 的 OCR 低置信答案被识别成 D，但回看原解析页后确认参考答案是 C。
- 我做错了什么：一开始考虑过把部分 `dirty_ocr_bracket` 低置信答案批量升级；如果这么做，会把错答案直接放进用户训练。
- 用户如何纠正：用户反复强调“千万不要把答案写错了，这是很严重的错误”，并要求必要时直接看视觉/原图保证质量。
- 根因：低置信 OCR 不是“格式不干净但大概率对”，而是可能把参考答案字符识别错；尤其扫描解析页里有批注、页脚、下一题残片时，单靠正则非常危险。
- 以后必须这样做：低置信答案只能进入后台修复队列；只有原解析页、原图视觉模型或明确人工视觉核验确认后，才能写入 `verified_repairs.json` 并重新跑同一套门禁。
- 可验证的防复发动作：`approved_seed` 中 `answer_confidence` 必须为 `high`；`answer_source=dirty_ocr_bracket` 不得进入前台；每个从低置信改高置信的答案必须在 `verified_repairs.json` 记录来源 PDF 和页码。

## 2026-07-10 · 600 题修复不能靠逐页手工慢看作为主流程

- 场景：数量关系 600 题已经生成 417 道视觉/原图待修队列，但我继续偏向逐页人工查看和小批量修复，用户等了很久后指出“太磨蹭了，我等你一个晚上你也跑不完”。
- 我做错了什么：没有及时把主流程切换成批量视觉执行、逐题落盘、失败可重试、严格导入，而是让人工视觉抽查承担了过多生产任务。
- 用户如何纠正：用户明确要求在保证成本的情况下保证质量和速度，并指出必要时直接视觉模型，不应该让修复拖一晚上。
- 根因：把“我能看图”误用成“我逐页手工看图”，没有区分小样本核验和 417 道批量生产；也没有第一时间实现并发、续跑和失败隔离。
- 以后必须这样做：600 题这类批量加工，主流程必须是机器批处理；人工视觉只用于抽查、争议题、图形裁剪和答案冲突核验。
- 可验证的防复发动作：视觉批处理脚本必须支持 `--limit=all`、`--concurrency`、`--retries`、默认跳过已有结果和单题失败日志；导入脚本必须只导入成功 JSON，失败日志不能进入 `verified_repairs.json`。

## 2026-07-10 · 图形题不能用 AI 重画替代原图

- 场景：数量关系 600 题中有 48 道图形依赖题，用户明确指出“有图的题目，你别就自己画，我担心你画的和题目不一样，我要原图”。
- 我做错了什么：前面只强调图形题必须视觉/原图核验，但没有把“上线展示也必须使用原 PDF 原图裁剪”写成硬规则，容易让后续实现误以为可以用 AI 或前端根据文字重新画图。
- 用户如何纠正：用户明确要求有图题必须保留原图，不能自己画。
- 根因：把“识别图形条件”和“展示图形条件”混在一起。识别时可以用视觉模型辅助，但产品展示和题目条件必须可追溯到原始题本。
- 以后必须这样做：图形依赖题上线前必须从原 PDF 裁剪题目原图，记录裁剪资产路径、PDF 来源和页码；AI 可以解释图，但不得重画图替代原图。
- 可验证的防复发动作：任何带 `image_dependent_question` 的题，如果没有原图裁剪资产，不得进入 `approved_seed`；前端渲染图形题时只能读取原图资产，不能读取 AI 生成图或手绘 SVG 作为题干图。

## 2026-07-10 · 原图裁剪只用于题干图形，不是整页截图入库

- 场景：我把“图形题要保留原图”说得不够精确，用户纠正“有图的题目才是贴进去，不是说别的没图的你贴整个题目和解析还是答案什么的进去”。
- 我做错了什么：前面容易让人误解成：只要有 PDF 原图，就可以把整页题目、解析或答案截图作为前台内容。这会破坏结构化题库，也会让用户端变成 PDF 看图而不是训练系统。
- 用户如何纠正：用户明确说明只有题目中依赖图形的部分需要贴原图，其他题干、选项、答案、解析仍应结构化录入。
- 根因：没有区分“后台核验页图”和“前台题干媒体资产”。后台可以看整页 PDF 辅助修复；前台只能展示作答必要的局部图形裁剪。
- 以后必须这样做：普通文字题只入结构化文本和标签；图形依赖题只额外保存题干所需的局部图形/表格/路线/统计图裁剪，不保存整页题目、整页解析或答案截图给用户。
- 可验证的防复发动作：题库 schema 的 `media` 只能表示 `question_figure_crop` 等题干局部资产；视觉提示词和 README 必须明确禁止把整页 PDF、解析页、参考答案截图当作前台题目内容。

## 2026-07-10 · 字段完整不代表题干和解析匹配

- 场景：数量关系题库清洗时发现部分候选题题干、选项、答案、题型字段看似齐全，但题干来自一道题，解析和答案来自同套另一道题。
- 我做错了什么：前面的 approved 门禁主要检查题干完整、四选项、答案高置信、题型干净，没有专门验证“题干和解析是不是同一道题”，导致 188 道高置信种子题中可能混入错配风险。
- 用户如何纠正：用户多次强调答案不能错、不能让脏题进入用户训练；本轮排查把这个风险具体化为题干/解析错配。
- 根因：扫描 OCR 按题号切块时会被页眉、批注、断行、题号残片干扰，同一套内解析块可能错位；字段齐全只是格式正确，不等于语义配对正确。
- 以后必须这样做：题库入 `approved_seed` 前必须检查题干与解析题头的相似度；疑似错配题打 `analysis_stem_mismatch` 留后台，不得前台可见。
- 可验证的防复发动作：`approved_seed` 中 `analysis_stem_mismatch` 数量必须为 0；自动相似度重配的题必须打 `analysis_auto_rematched_needs_review`，不能直接进入 `approved_seed`，必须经原图/视觉模型或人工核验后再通过同一套门禁。

## 2026-07-10 · 题库生产线不能把 OCR 和视觉模型对立起来

- 场景：数量关系 600 题已通过 OCR 形成候选题，用户追问“你不是有视角模型吗，为什么要用 ocr，是担心额度吗”，随后确认目标是在保证成本的情况下保证质量和速度。
- 我做错了什么：前面解释 OCR 流程时，容易让人理解成“只做 OCR，不用视觉模型”。这会误导产品方案：图形题、版面题、缺选项、缺答案题本来就不该只靠 OCR。
- 用户如何纠正：用户明确指出应考虑视觉能力，不要把 OCR 当成唯一方案。
- 根因：没有把后台题库生产线拆成“OCR 批量打底”和“视觉/原图修复疑难题”两个阶段；成本控制和质量控制的关系讲得不够具体。
- 以后必须这样做：批量题库导入默认采用分层策略：OCR 负责全量低成本打底，规则门禁负责挡脏题，视觉模型/原图回看负责修复 OCR 低置信、图形依赖、题干选项污染和答案缺失题。
- 可验证的防复发动作：题库导入产物必须包含视觉/原图修复队列，记录每道待修题的问题、优先级、应回看的 PDF 页、修复目标和是否允许前台训练；图形依赖题必须 `blocked_from_user_training=true`，不得绕过门禁进入 `approved_seed`。

## 2026-07-10 · 题库导入不能把用户变成人工 OCR 审校员

- 场景：数量关系 600 题已经通过 OCR 生成候选题，但严格门禁后只有一小部分能直接给前台 demo 使用；用户指出“整个 600 题怎么办”“这些题目你都得自己弄好，我不会帮你看”。
- 我做错了什么：最初把“前台只接入高置信题”讲得像最终方案，容易让用户理解成剩余题被落掉，且没有第一时间把剩余题的修复队列、问题分布和后台处理责任说清楚。
- 用户如何纠正：用户明确要求 Agent 自己把题库整理好，不要让用户一题题看，因为用户审题效率太低。
- 根因：题库生产线和用户端训练题库没有分层表达清楚。`approved_seed` 只是可上线种子题，不等于全量题库完成；`needs_review` 也不是丢弃，而是后台修复队列。
- 以后必须这样做：扫描题库导入必须分为 `raw` 原始资产、`clean_candidates` 结构化候选、`approved_seed` 用户可见题三层；任何一层数量变化都要说明清楚。用户端只读取通过硬门禁的题，剩余题由 Agent 通过批量规则、审核页、抽样验收和必要的原图回看继续修复。
- 可验证的防复发动作：最终回复和状态文档必须同时写出全量题数、前台可用题数、待修题数、主要问题分布；不得让用户承担逐题 OCR 校对；不得把带 OCR 脏题干、脏选项、脏题型或低置信答案的题放进用户训练流。

## 2026-07-05 · 正视指标通过不等于用户肉眼觉得清楚

- 可验证的防复发动作：Phase 4 新增申论题库、错题追踪、Skill 落地节点；Phase 6 新增 GitHub 干净总装节点，要求所有剩余文件归入“正式纳入/继续开发/历史归档”，禁止 `git add -A` 混入数据库、密钥、临时产物或未验收半成品。

## 2026-07-13 · AI 教练必须同时支持自由提问和可信上下文，不能二选一

- 场景：Phase 5 初版服务端为了防止伪造题目事实，要求所有学科对话必须带 `activity_id`；这虽然安全，却会让用户从独立“问老师”入口进入时无法新建对话。
- 根因：把“上下文必须可信”错误实现成“任何提问都必须有作答上下文”，忽略了产品已经明确存在自由咨询和训练追问两条路径。
- 以后必须这样做：无 activity 的独立提问允许进入版本化方法 Skill，但上下文明确为 `standalone_method_question`、证据为空，不能据此生成客观弱项；从训练页进入只传服务端引用，后端再按 JWT 读取垂直记录，客户端摘要、答案和 `user_id` 一律不可信。
- 可验证的防复发动作：API 和浏览器都要覆盖“无记录自由提问成功且0问题卡”“本人已完成活动带上下文成功”“他人/伪造 activity 拒绝”“未交卷不泄露答案”四条路径。
