# 错误复盘

本文件记录 AI 在本项目中被用户纠正过的错误，以及以后必须执行的防复发动作。
不要只写“下次认真”，每条都要能指导后续验收。

## 2026-07-05 · 正视指标通过不等于用户肉眼觉得清楚

- 场景：动态解题页已经读到 `sectionFacing=1.000`，但用户继续指出 3D 里的截面不像正面朝向自己，小图卡片也不该留在舞台里；点选项后还会把下方内容顶开。
- 我做错了什么：上一轮把“相机方向指标接近 1”和“小图可显示”当成主要验收，没有同时做肉眼截图、点击 A/B/C/D 的布局矩阵、移动端宽度检查。
- 用户如何纠正：用户明确说“那个正面那个地方还是没有朝向我”“正对截面你放一个小图在那里干吗”“没点一个选项，图就会放大，把东西顶下去了”。
- 根因：验收维度太单一。空间方向、视觉清楚度、布局稳定性是三件事；一个数值通过不代表用户第一眼能看懂，也不代表交互后布局不跳。
- 以后必须这样做：涉及教学 3D/截面/视图方向时，必须同时验收：数据指标、真实浏览器截图、点击矩阵、桌面和手机响应式、控制台日志。
- 可验证的防复发动作：动态解题页最终验收必须读取 `frontCardExists=false`、`sectionFacing>=0.95`、初始 `selected=null`，并逐个点击 A/B/C/D，记录 3D 舞台和下方截面区的 `y/height` 是否稳定；不得再把 `front-section-card` 作为通过条件。
- 本次补救：`LESSON-015E` 删除 `#front-section-card`，把正面截面放回 3D 本体；桌面 1440×1000 下 A/B/C/D 点击后 3D 舞台保持 `y=174 height=520`，移动端 390×844 下保持 `x=11 y=839 width=368 height=470`。

## 2026-07-05 · 不能把正面截面浮层当成 3D 正视

- 场景：动态解题页已经加了“正对截面”浮层，但用户指出 3D 舞台里的真实橙色截面仍然像在背后，不是截面正面面向用户。
- 我做错了什么：只验证了右下角 SVG 浮层有真实轮廓，没有验证 3D 相机是否沿切面法向看过去；结果学生看 3D 本体时仍然需要从斜侧面猜。
- 用户如何纠正：用户明确说“我要的是截面正面面向我，而不是在背后”。
- 根因：把“另有一个正面对比图”误当作“模型里的真实截面正面对用户”；缺少可读指标证明相机视线和切面法向接近一致。
- 以后必须这样做：涉及截面正视、投影视图、左右视图、俯视图这类空间方向问题，不能只看截图或 DOM，必须暴露并读取相机/法向/视图方向指标。
- 可验证的防复发动作：动态解题页浏览器验收必须读取 `#lesson-canvas[data-section-facing]`，正式题初始状态应接近 1；若低于 0.95，不能说“3D 截面正对用户”。即使该指标通过，也必须配合截图和点击矩阵，确认 3D 本体清楚、布局不跳。

## 2026-07-05 · 草稿题答案不能等用户一张张替我核

- 场景：用户连续给出多个参考视频题截图答案，并指出“你不是能看视频吗，还要我去确认”，要求我先把大致答案写好，再让用户最终确认。
- 我做错了什么：之前只把视频题作为 `draft-unverified` 题图入口，没有把用户已经明确说出的答案记录下来，也没有把与正式题冲突的说法标成待复核。
- 用户如何纠正：用户给出第二张 A、第三张 C、第四张 D、第五张 D、第六张 B、第七张六边形，并要求后续可以直接填答案或让我自己先写候选答案。
- 根因：把“未正式人工核验”误解成“答案字段完全不录入”；这样会让后续 Agent 又从零开始问用户。
- 以后必须这样做：草稿题可以保留 `draft-unverified`，但用户已给出的答案必须进入结构化字段，状态标明 `user-provided-needs-review`；与现有正式答案冲突时写 `conflict-needs-review`，不能默默覆盖。
- 可验证的防复发动作：`draft-video-questions.json` 中应能直接查到用户给出的候选答案；专项测试必须覆盖草稿答案入库和正式题冲突保留。

## 2026-07-05 · 动态解题页不能只把截面放在侧面或下方

- 场景：用户指出动态解题页 3D 舞台里一直看不到截面，截面应该正面给用户看见；第二道题源图也不应该保留整段视频画面。
- 我做错了什么：最初只加了“正对截面”浮层和题图裁剪，但浏览器验收发现浮层仍是“等待切面”；随后又暴露出无切面关键帧返回值结构不一致导致页面载入失败。
- 用户如何纠正：用户明确说“不用等确认，继续完成”，要求我自己判断并验收这些剩余问题。
- 根因：把“页面有一个浮层”误当成“学生一进来能看见真实截面”；同时没有第一时间在真实浏览器里检查初始状态、资源版本和错误提示。
- 以后必须这样做：动态教学页验收不能只看 DOM 或文字，必须在浏览器里检查初始状态：正式题加载后不能载入失败，真实截面必须在 3D 舞台内有正面轮廓，且不提前选中 A/B/C/D。
- 可验证的防复发动作：涉及前端模块改动后必须同步提升静态资源版本号；浏览器验收要读取 `engineStatus`、`frontCardExists=false`、`#lesson-canvas[data-section-facing]`、实时截面 `.section-shape`、题图 `naturalWidth/naturalHeight`、选项选中数量、点击矩阵和控制台 error/warn。
- 本次补救：`LESSON-015C` 新增正对截面浮层，正式题加载后自动显示一个真实切面预览但不预选答案；第二题题图裁成 996×430；浏览器验收确认 `engine=真实截面 · 1 个轮廓`、浮层有真实轮廓、选项选中数为 0、控制台无 error/warn。

## 2026-07-05 · 把本地服务断开误说成浏览器拦截

- 场景：用户打开 `three-view-training.html` 时，页面无法访问，我上一轮把问题笼统说成“浏览器安全策略拦截”。
- 我做错了什么：没有第一时间确认 `8089` 静态服务是否还在监听，也没有先用 `curl -I` 区分 `ERR_CONNECTION_REFUSED` 和真正的浏览器自动化限制。
- 用户如何纠正：用户指出不能每次都说被拦截，必须自己解决；用户真正关心的是页面能不能打开、能不能验收。
- 根因：把“自动化工具不能操作错误页”和“用户页面打不开”混在了一起，没有先排查最基础的服务状态。
- 以后必须这样做：遇到本地页面打不开，先跑 `lsof -nP -iTCP:端口 -sTCP:LISTEN` 和 `curl -I 'URL'`。如果是 `ERR_CONNECTION_REFUSED`，先启动服务；只有服务返回 200 后，才进入浏览器自动化验证。
- 可验证的防复发动作：静态训练页统一用 `npm run dev` 启动 `127.0.0.1:8089`；任务记录里必须写清楚服务状态、HTTP 状态和浏览器验证三者分别是否通过；`npm run dev` 必须能自检端口并后台启动，不能只依赖前台终端挂着。
- 本次补救：`THREEVIEW-003` 验收前已分别记录 `lsof` 监听、`curl -I` 200、浏览器整组训练和控制台日志；`DEVSERVER-002` 又新增 `tools/dev-server.mjs`，让 `npm run dev` 在端口未开时自动后台启动、端口已开时复用，并提供 `npm run dev:status` 查看状态。

## 2026-07-05 · 前台启动服务不等于稳定可验收服务

- 场景：用户再次打开 `three-view-training.html?verify=orientation-fix` 失败，排查发现 `8089` 没有任何进程监听。
- 我做错了什么：上一轮虽然写了 `npm run dev`，但它仍是前台 `python3 -m http.server`；一旦终端或会话结束，服务就会停。这个方案不能支撑用户持续验收。
- 用户如何纠正：用户指出“这个问题存在好多次了，为什么老是出现，要解决”。
- 根因：把“一次性启动成功”误当作“稳定可用”。本地开发页需要有自检、后台启动、PID/日志和状态查询，否则验收链路不可靠。
- 以后必须这样做：凡是交付本地网页验收，必须同时提供可重复运行的启动命令和状态检查命令；如果服务是验收前提，先证明端口监听与 HTTP 200，再让用户看页面。
- 可验证的防复发动作：`npm run dev` 必须能在服务未开时启动，在服务已开时复用；`npm run dev:status` 必须返回 `open=true`；运行痕迹写入 `.codex-runtime/` 并加入 `.gitignore`。
- 本次补救：新增 `tools/dev-server.mjs`，`npm run dev` 改为 ensure 模式；验证 `pid=72457` 正在监听，`curl -I` 返回 `HTTP/1.0 200 OK`。

## 2026-07-05 · 页面验证不等于数学事实验证

- 场景：基础截面页修正正方体/长方体“直角三角形”时，我上一轮说“验证过”，但实际主要验证了按钮、DOM、默认 3D 状态和实时文案。
- 我做错了什么：把页面验收说得像数学事实验收一样完整，没有把“盒体三角截面是否可能被误判为直角三角形”做成单独验证。
- 用户如何纠正：用户指出“你不是说验证过吗”，要求把这类错误写进项目复盘，避免下次再犯。
- 根因：验证层级混淆。UI 状态正确只能证明交互接线和默认状态，不等于证明几何事实、算法边界和拖动过程都正确。
- 以后必须这样做：涉及几何、数学、法律、医学、金融等领域事实时，最终说明必须区分“页面验证 / 算法验证 / 领域事实验证 / 用户体验验证”。
- 可验证的防复发动作：几何结论必须至少有一个确定性样例、抽样扫描、函数测试或推导证据；不能只点击页面默认状态就宣称“事实验证通过”。
- 本次补救：`section-foundation.js` 已让正方体/长方体三角截面分类不再因接近 90 度被误判成“直角三角形”；`tests/reasoning-lesson-layout.test.mjs` 已加入盒体禁用直角误判的源码断言。

## 2026-07-05 · 局部验证不能替代全量截面矩阵

- 场景：基础截面页已经有 3D、实时截面和缩略图，但用户继续发现“长方体三角形实际是五边形”等问题。

- 用户如何纠正：用户要求“不要让我去验证”，明确指出要先验证旁边小图是否符合标签，再验证右侧 3D 是否呈现对应截面。
- 根因：验收样本太窄。一个页面上有 44 个截面卡片，只测其中几个会让错误预设继续混在页面里。
- 以后必须这样做：涉及基础图形库、题库选项、模板库这类“多卡片/多样例”页面时，必须跑全量矩阵验收；每个卡片都要检查标签、预览、主视图和实时结果是否一致。
- 可验证的防复发动作：新增或调整截面预设时必须运行 `tests/foundation-section-presets.test.mjs`，并在浏览器自动点击所有 44 个基础截面卡片，失败项不得靠肉眼跳过。
- 本次补救：`LESSON-014N` 将 7 个错位预设改为真实交线参数，并新增真实几何矩阵测试；浏览器 44/44 卡片验收通过。

## 2026-07-05 · 三视图题面投影和 3D 固定相机必须同源验证

- 场景：三视图训练页中，题面左视图/右视图是按题库规则生成的，但右侧 3D 模型点击“左视图/右视图/俯视图”后，视觉方向和题面网格不一致。
- 我做错了什么：之前只验证了题库投影自洽、按钮可点击和模型非空，没有验证“3D 固定相机看到的左右/上下/遮挡顺序”和同名二维视图规则一致。
- 用户如何纠正：用户指出必须以正视图为基准，转到左边是左视图、转到右边是右视图、转到上面是俯视图；不能让题面视图和右侧 3D 切换视图方向各说各话。
- 根因：`VIEW_RULES` 与 `setCamera()` 分开维护，且题库坐标映射到 Three.js 坐标时没有显式处理 x 轴镜像；俯视图还沿用了从下往上的遮挡顺序。
- 以后必须这样做：凡是做三视图/投影训练，题库投影规则、3D 坐标映射、固定相机位置必须由同一套配置证明一致；不能只靠肉眼看模型，也不能只测题库 JSON 自洽。
- 可验证的防复发动作：`tests/three-view-training.test.mjs` 必须包含“fixed 3D view cameras use the same orientation as the 2D view grids”测试，逐个验证主/左/右/俯的屏幕横向、纵向和深度遮挡顺序。
- 本次补救：新增 `VIEW_CAMERA_POSES`，训练页固定视角读取同源配置；3D 坐标入口统一镜像 x 轴；俯视图改为从上往下看；浏览器逐个点击主/左/右/俯/自由按钮并读取 `data-camera-position` 通过。

## 2026-07-05 · 选后验证区不能在未作答前暴露答案模型

- 场景：三视图训练页文案写着“选完马上看立体摆法”，但页面未选择 A/B/C/D 时已经显示真实 3D 模型。
- 我做错了什么：前一轮主要验收了布局、视角按钮、模型方向和答题后的投影视图，没有把“未作答状态是否泄露验证模型”列成必测项。
- 用户如何纠正：用户指出“选完看效果是不是得选完啊，没选也能看见 3D，是不是逻辑有问题”。
- 根因：把 3D 模型当成页面辅助区默认渲染，没有按训练流程区分“做题阶段”和“验证阶段”；验收只覆盖答后可用，没有覆盖答前禁止。
- 以后必须这样做：凡是“选后展示答案、答后验证、提交后讲解”的训练页，验收必须同时检查未作答状态和作答后状态；未作答时不能显示会直接暴露答案结构的模型、解析、投影或统计。

- 本次补救：`THREEVIEW-007` 新增 `model-gate` 选前遮罩与 `setModelAccess()` 统一锁定/解锁；专项测试覆盖选前 CSS 和 JS 接线；浏览器验收确认选前隐藏、选后显示。

## 2026-07-05 · 几何可行不等于候选图完全对上

- 场景：动态解题页验证四棱锥与圆柱组合截面时，候选 C 属于“可以切出这一类曲边+直边截面”，但页面文案和布局容易让用户理解成“候选图已经和真实截面完全对上”。
- 我做错了什么：把“截面类型可行”和“当前候选简图与真实 3D 截面完全匹配”混在一起；下方真实截面图还使用独立的切面局部坐标，导致它和用户正在看的 3D 方向不一致。
- 用户如何纠正：用户指出“逐项验证哪里对上了，没对上说对上了”，并要求先选择答案再解析，且 3D 和下方截面图必须是同一个方向。
- 根因：教学判断缺少层级。几何引擎能证明某类截面可出现，但考试选项还要继续核对方向、比例、曲边朝向和连接位置；UI 若提前暴露解析，也会破坏学生先判断的流程。
