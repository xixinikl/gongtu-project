# 公途 AI 协作章程

> 版本：2.0
> 生效日期：2026-06-29
> 适用范围：所有 AI 协作者及其参与的代码、文档、测试与发布工作。

## 第一章：唯一技术栈

空间几何与工图推理模块必须在现有项目上增量建设，不另起一套应用：

| 层级 | 唯一选择 | 用途 |
|---|---|---|
| Web 交互 | 原生 HTML、CSS、JavaScript | 延续现有前端，降低学习和部署成本 |
| 3D 渲染 | Three.js | 模型显示、相机、灯光、交互与切割可视化 |
| 几何运算 | Three.js 几何算法 + three-bvh-csg | 平面求交、组合体和截面计算 |
| 后端 | Python + FastAPI | 复用现有 API 服务 |
| 实时通信 | FastAPI WebSocket | AI 建模任务进度和错误反馈 |
| 数据库 | SQLite | 复用现有题库与用户数据方案 |
| AI 接口 | 可配置的 OpenAI 兼容多模态接口 | 图片与文字解析为结构化模型草稿 |
| 模型协议 | 版本化 Geometry JSON | 隔离 AI、题库和 Three.js，禁止 AI 直接生成可执行代码 |
| 桌面端 | Electron | 复用现有桌面外壳 |
| CI | GitHub Actions + Ruff + mypy + Bandit + pytest | 静态检查、安全检查和自动测试 |

选择理由：该组合复用现有工程、部署便宜、初学者可维护，并原生支持 WebSocket。未经架构任务记录和用户批准，不得并行引入 React、Vue、另一套后端或另一种数据库。

## 第二章：规则优先级与版本化

1. 用户当前明确指令优先于仓库文档。
2. 本文件是 AI 执行工作的最高仓库规则。
3. `doc/PROJECT_RULES.md` 是通用开发规范；与本文件冲突时以本文件为准。
4. `README.md` 只描述产品和使用方法，不承担行为约束。
5. 新需求出现时，应修改规则版本和变更记录，不得口头覆盖后继续执行。
6. 规则变更本身必须作为 `TASKS.md` 中的独立任务接受检查、提交和推送。

## 第三章：任务看板铁律

1. 所有工作必须先进入 `TASKS.md`，没有任务编号不得修改业务代码。
2. 每个任务必须足够小，单次提交最多修改 3 个文件。
3. 每个任务只允许一个可验证结果；实现、测试、文档过大时必须继续拆分。
4. 状态统一使用：
   - `○`：待开始，对应 `- [ ]`
   - `◐`：进行中，对应 `- [ ]`
   - `●`：已完成，对应 `- [x]`
   - `⛔`：阻塞，对应 `- [ ]`
5. 同一时刻只能有一个 `◐` 任务。
6. 开始任务前，先将其状态改为 `◐` 并在 `CURRENT_STATUS.md` 记录计划。
7. 完成任务后，只有通过验收命令才能改为 `- [x] ●`。
8. 不得批量补勾、先勾后测或把未完成工作描述为完成。

## 第四章：Git 提交与上交铁律

1. 稳定分支为 `main`，任何开发不得直接提交到 `main`。
2. 一个完整功能使用一个 `feature/功能短名` 分支；本模块使用 `feature/spatial-geometry-lab`。
3. 每完成一个 `TASKS.md` 任务，必须立即产生一个独立 Git 提交。
4. 提交信息必须与任务中状态标记之后的任务名精确一致。
5. 每次提交前必须检查 `git diff`，不得夹带无关文件或用户未提交的改动。
6. 每个任务提交后必须推送当前功能分支，让远端可检查。
7. 功能完成后必须通过 Pull Request、CI 和人工验收，才能合并到 `main`。
8. 合并方式使用 squash 或保留提交历史，由 PR 阶段决定；禁止强推 `main`。
9. 遇到已有脏工作区、分支分叉或冲突，必须先保护现有成果，不得使用 `reset --hard`。

## 第五章：可见性铁律

每次任务状态变化都必须更新 `CURRENT_STATUS.md`，至少记录：

- 当前分支与任务编号
- 刚刚完成了什么
- 实际修改的文件
- 执行了哪些验收命令及结果
- 下一个要做什么
- 已知风险或阻塞
- 对应提交哈希和远端推送状态

任何“完成”都必须能够由文件、命令输出和提交三者共同证明。

## 第六章：质量门禁

任务完成前按影响范围执行：

1. 文档任务：检查 Markdown 结构、任务状态一致性和 Git diff。
2. Python 任务：Ruff、mypy、Bandit、相关 pytest。
3. JavaScript/Three.js 任务：语法检查、浏览器启动检查和相关交互测试。
4. API 任务：正常请求、错误请求和权限边界测试。
5. 几何算法任务：固定数学样例、浮点误差边界和退化情况测试。
6. UI 任务：桌面与窄屏检查，必要时保留截图证据。
7. 全阶段完成：运行完整 CI 等价检查。

测试失败时任务保持 `◐` 或改为 `⛔`，不得提交“完成”状态。

## 第七章：几何正确性铁律

1. AI 只负责把图片和文字转换为 Geometry JSON 草稿，不得直接决定数学答案。
2. 截面、投影、相交和面积必须由确定性几何引擎计算。
3. 每个 AI 模型必须经过 Schema 校验、几何校验和人工预览确认后才能进入正式题库。
4. 题目原图信息不足时必须标记“不确定”，禁止虚构被遮挡的结构。
5. 基础模型、积木组合、旋转体和自由组合体必须使用版本化参数表示。
6. 每道正式题必须保留原图、文字、模型版本、标准答案和验证状态。

## 第八章：安全与零硬编码

1. 所有密钥必须通过 `.env` 读取。
2. 仓库只允许提交 `.env.example`，不得提交真实 `.env`、Token 或用户隐私。
3. 上传图片必须限制类型、大小和文件名，并与可执行文件隔离。
4. AI 输出一律视为不可信输入，必须解析和校验，禁止执行其中代码。
5. 数据库变更必须可迁移、可回滚，并在测试数据库先验证。
