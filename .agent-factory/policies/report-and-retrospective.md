# 公途每日验收报告与错误复盘规则

## 两份报告

每日任务必须产出两份不同读者的内容:

| 文件 | 读者 | 用途 |
|---|---|---|
| `codex-summary.md` | Codex | 压缩事实输入，降低 token 消耗 |
| `daily-report.md` | 用户 | 人类可读日报，说明今天是否安全、哪里失败、下一步做什么 |

`codex-summary.md` 可以偏机器化；`daily-report.md` 必须给人看得懂，不要求用户读原始日志。

## 用户日报必须回答

1. 今天跑了哪些检查。
2. 总结论是 `pass`、`conditional` 还是 `fail`。
3. 如果失败，失败的是哪条检查，影响什么能力。
4. 是否尝试自动修复；修复是否通过复验。
5. 哪些内容没有覆盖。
6. 下一步是自动开 PR、等待人工确认，还是只需观察。

## 什么进入仓库错误复盘

不要把每天的流水账提交进仓库。只有满足以下任一条件，才写入 `doc/retrospectives/`:

- 同类错误重复出现 2 次及以上。
- 错误暴露了项目规则缺口、CI 缺口或测试盲区。
- 修复方式可复用，能避免未来 AI 或人重复踩坑。
- 涉及跨平台、鉴权、数据、启动脚本、AI 批改、题库等高风险区域。

## 什么不进入错误复盘

- 一次性拼写错误。
- 纯格式化问题。
- 依赖网络抖动。
- 已在报告里说清且没有复用价值的红灯。
- 包含密钥、用户数据或敏感日志的内容。

## 复盘文件格式

文件名:

```text
doc/retrospectives/YYYY-MM-DD-short-slug.md
```

内容:

```markdown
# 错误复盘: 标题

## 触发

## 影响

## 根因

## 修复

## 防复发规则

## 相关检查
```

## 自动修复边界

公途每日任务只能自动修复低风险 Ruff 问题。其他问题最多创建修复分支和 PR，不得直接推 main。
