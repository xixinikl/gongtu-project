## 0. 一句话目标

以 `verbal-ai-v3` 已确认的公途黑金、宣纸、古风 SVG 界面为统一学习外壳，将言语、数量、平面图推、立体图推、申论和 AI 教练接入同一个 `users.id` 和统一学习证据合同，完成从训练、保存、复盘、追问到下一步任务的真实闭环。

本 Goal 不是“把几个 Demo 链接起来”，而是将已有专业板块按一套用户、记录、导航、AI 上下文和验收标准总装成可继续维护的产品。

## 2. 产品与工程原则

1. **一个产品外壳**：模块界面可因任务密度不同，但品牌、导航、字体、边框、图标语法和返回路径一致。
2. **一个用户主键**：只使用现有 `users.id`；所有用户数据归属从 JWT 获取。
3. **一份模块清单**：模块 ID、名称、入口、Skill 和证据类型必须有单一事实源。
4. **客观数据先于 AI**：正确率、用时、题型、改选、跳过由程序计算；AI 只做解释、归纳和任务规划。
5. **用户语言与内部语言分离**：`confidence / strong evidence / weak evidence` 可作内部字段，前台改写为“多次出错 / 还需要几题确认 / 已有改善”。
6. **专业能力不硬套**：言语看问法、结构和选项；数量看题型、取舍、方法和时间；图推看规律和空间验证；申论看要点、结构和表达。
7. **正式数据不被 AI 改写**：题目、选项、答案、官方解析、词库例句和申论原答为事实层。
8. **每阶段一个 PR**：六个阶段各自独立分支、独立变更集和独立验收证据；禁止用一个巨型 PR 混合六阶段。
9. **不因普通中途反馈丢主线**：反馈先归类到当前阶段、下一阶段或 backlog；当前阶段未过门禁不得跳走。
10. **安全和事实冲突可以中止**：越权、丢数据、答案冲突、迁移不可回滚或密钥泄露时必须停在当前阶段。

## 3. 范围与非目标

### 3.1 必须完成

- 将 V3 视觉和信息架构转为正式外壳，保留现有页面精调多次的字体、边框、色板和图标语法。
- 重做“我的复盘”和“套题实战”，解决内部术语、功能空洞、排版过短和大量无效留白。
- 建立统一学习活动、作答、问题、任务和 AI 运行合同，并全部归属 `users.id`。
- 接入 801 成语词库、231 题逻辑填空、已完成的 30 套 600 题片段阅读和最终完成的数量题库。
- 在正式 600 题片段阅读及后续合格言语题库中，保留“题中成语/词语悬停查看解释、例句、人民网链接，并加入今日任务或收藏/待复习”联动。
- 接入平面图推、四段立体图推主链和申论批改摘要。
- AI 教练支持从各板块带上下文进入、自由提问、按模块调用 Skill、形成问题卡和下一步任务。
- 完成未登录、登录、刷新、登出重登、A/B 用户隔离和 AI 失败降级的真实路径验收。

### 3.2 明确不做

- 不新建第二套 UC ID、账号系统或题库用户表。
- 不在本 Goal 内更换前端框架或数据库。
- 不因为总装而重写已验证的 3D 几何内核。
- 不把 `reasoning-lesson.html` 放入立体图推固定主导航。
- 不在数量最终交接前猜测数量题数、答案、标签或 Skill 能力。
- 不在本 Goal 接入历史题库中的 219 题片段阅读；本次片段阅读以已完成的 600 题为权威库。
- 不把 AI 生成的逻辑填空讲解标成“官方解析”；原解析缺失时必须明确告知用户。
- 不使用假统计、假 AI 诊断或访问页面次数伪造学习效果。
- 不在未通过真实用户隔离前宣称“统一用户数据已完成”。
- 不自动合并 PR 或部署生产；合并和部署由用户或仓库流程授权。

```text
公途
├─ 言语
│  ├─ 词语积累
│  ├─ 逻辑填空
│  ├─ 片段阅读
│  ├─ 套题实战
│  └─ 我的复盘
├─ 数量关系
│  ├─ 今日训练
│  ├─ 单题诊断
│  ├─ 套题策略
│  ├─ 系统学题型
│  └─ 我的复盘
├─ 图形推理
│  ├─ 平面规律
│  ├─ 立体空间
│  └─ 我的复盘
├─ 申论批改
└─ AI 教练
```

顶部主栏使用同一套古风 SVG、高度、间距、边框和当前态。各板块的次级导航使用相同组件语法，但不硬套相同功能。

### 4.2 重做后的“我的复盘”

```text
本周复盘
├─ 最近容易错在哪里（用户语言）
├─ 这个问题怎么形成（题号 / 用时 / 选择作为证据）
├─ 下一步做什么（可直接开始的任务）
├─ 练完有没有改善（前后对比）
└─ 近期训练时间线与已完成记录
```

- 前台禁止直接显示“强证据 / 待补证据 / confidence”。
- 当数据不足时显示“再做 5 题，才能判断是否稳定”，不伪造弱项。
- 页面高度由真实内容与时间线构成，不用空卡片硬撑，也不留下无功能的大块空白。

### 4.3 重做后的“套题实战”

```text
创建本次实战
├─ 模式：言语专项 / 数量专项 / 行测混合
├─ 题型构成和题量
├─ 计时：正计时 / 倒计时
├─ 建议顺序与跳题规则
└─ 开始实战

作答中
├─ 题号进度
├─ 用时和剩余时间
├─ 标记 / 跳过 / 返回
└─ 交卷

交卷后
├─ 正确率与各题型分布
├─ 用时分布
├─ 应该放弃但没放弃的题
├─ 本次可以改进的做题顺序
└─ 下一组训练
```

### 4.4 AI 教练

- 保留已确认的三栏结构：模块选择、对话与工具、当前模块问题本。
- 使用公途统一视觉 token，但不为表面一致破坏对话密度。
- 从题目页进入时携带 `module / activity / question / answer / timing / official analysis / skill / return_url`中适用字段。
- 自由提问时先选模块；不确定时进入学习规划路由。
- 问题本保存可继续训练的结论，不把全部闲聊原文堆进复盘。

### 4.5 必须保留的成语卡与题中词语体验

**成语卡不允许重做成普通列表或新风格卡片**。必须保留用户已长期调整的原卡片层级：

```text
卡片正面：成语 + 学习状态
卡片翻面：
├─ 成语解释
├─ 例句（来自用户 Excel，保留换行）
└─ 人民网搜索链接
```

必须保留的用户状态和操作：

- 今日任务、全部词库、易忘攻克。
- 全部、到期、易忘、收藏筛选。
- 收藏、加入今日任务、间隔复习、刷新和重登保留。
- 例句标题只写“例句”，不写“人民网例句”；人民网只是跳转搜索链接。

正式言语题库中的词语联动：

```text
悬停/键盘聚焦题中成语或词语
├─ 词语与来源
├─ 解释
├─ Excel 例句
├─ 人民网链接
├─ 加入今日任务
└─ 收藏/加入待复习
```

- 未收录词语必须明确标记“待补充”，可先收藏或加入任务，不由 AI 伪造词典释义。
- 这套交互能力来自词库层，将同时用于 231 题逻辑填空和 600 题片段阅读。

### 6.2 连续推进

- Goal 启动后，按 Phase 1 → 6 依赖顺序推进。
- 每个阶段内可拆 3—7 个小节点，同一时间只有一个节点进行中。
- 每个节点必须独立验证，不在阶段末一次性补写全部证据。
- 阶段门禁通过后才可创建该阶段 PR；PR 描述必须列出范围、验证、风险和未完成项。
- PR 的创建不等于自动合并。下一阶段的基线必须是上一阶段已审阅的确定代码，不在未审阅巨型脏分支上无限叠加。
- 用户页面反馈归入当前阶段验收时，在该 PR 内修正；不属于当前阶段则记入后续阶段，不丢失主任务。

### 6.3 六阶段后的 GitHub 干净总装

六个阶段全部通过各自门禁后，不直接把当前本地脏工作区整体提交。最终收口必须执行：

1. 按 Phase 1→6 的依赖顺序汇总已经验收的提交和 PR，不以文件时间或聊天顺序猜测。
2. 盘点两个 worktree 与所有相关分支的剩余未提交文件，逐项归入“正式纳入 / 继续开发 / 历史归档”。
3. 在干净总集成分支上汇总正式成果；不得用 `git add -A` 将临时文件、数据库、密钥、旧 Demo 或未验收半成品混入。
4. 对干净分支重跑完整回归、数据门禁、浏览器主路径和 A/B 隔离；结果必须与六个阶段证据一致。
5. 后续小改动回到所属阶段的修订提交或最终集成 PR，不要求用户现在一次定稿，也不覆盖已保全的原始成果分支。

当前已保全的本地成果包括：`cx/vocab-verbal-original-ui` 提交 `7ac5a1b`。保全提交只防止成果丢失，不等于已迁入正式总装。

## 9. 阶段门禁与停止条件

| 风险 | 防护 | 停止/回滚条件 |
| --- | --- | --- |
| 现有用户数据丢失 | 迁移前备份，可重复 migration，不删旧表 | 数量、用户或作答记录异常立即停止 |
| 用户越权 | JWT `users.id`，所有查询带用户约束 | 任一 A/B 隔离失败，后续阶段不得继续 |
| 题库答案倒退 | 题库权威源、导入报告、服务端判题 | 题目/答案/解析错位时停止该 adapter 接入 |

| AI 改写事实 | 官方内容由服务端注入并校验 | 回答与正式答案冲突时拒绝展示 |
| 密钥泄露 | 仅服务端 env，日志脱敏 | 任何密钥进入 Git/前端/截图则立即停止并轮换 |
| 外壳破坏专业页 | 专业页保留任务布局，只统一外壳与路径 | 3D 画布、刷题密度或申论长文体验倒退则回退外壳嵌入 |

## 10. 需求变更与不中断规则

用户在查看页面时提出反馈，Agent 必须先分类：

1. **阻塞当前阶段验收**：纳入当前 PR，修复后重跑门禁。
2. **属于后续已定义阶段**：写入该阶段节点，继续当前任务。
3. **新的长期扩展**：写入 backlog，不改变当前 Goal 完成定义。
4. **安全、数据或事实冲突**：立即停在当前节点，先修复或报告真实阻塞。

“不中断任务”的含义是不因普通视觉意见、新想法或中途提问丢失 Goal 主线；不意味着可以忽略越权、答案错误、丢数据或未审阅 PR 风险。
